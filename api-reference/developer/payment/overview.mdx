The **Payment API** allows you to create and retrieve payment transactions associated with your account.

This API enables:

- Creating payment links to collect funds
- Retrieving individual payment details
- Listing outgoing payments
- Listing incoming paymentsd

---

## What This API Provides

The Payment API allows your integration to:

- Generate secure payment links for wallet-based collections
- Retrieve a payment record by its unique identifier
- List payments sent from the user account
- List payments received into the user account

Each payment record includes:

- Unique payment ID
- Amount (USD)
- Payment status
- Payer and payee details
- Timestamps
- System-generated metadata

## Typical Integration Flow

<Steps>

  <Step title="1. Create a payment link">
    Call <code>Create Payment</code> with the amount and optional description to generate a shareable payment URL.
  </Step>

  <Step title="2. Share the payment link">
    Send the generated URL to the payer for completion.
  </Step>

  <Step title="3. Track payment status">
    Use <code>Get Payment</code> to retrieve transaction details and monitor status updates.
  </Step>

  <Step title="4. Reconcile transactions">
    Use <code>List Sent Payments</code> and <code>List Received Payments</code> for reporting and reconciliation.
  </Step>

</Steps>

---

## Pagination

The list endpoints require:

- `rows_per_page` (10, 20, 50, or 100)
- `page_no` (starting from 1)

Responses include:

- `records`
- `totalRecords`
- `queryGeneratedTime`

Use pagination to efficiently handle large transaction volumes.

---

## Best Practices

- Always validate amounts before submission.
- Store the returned `payment_id` for reconciliation and status tracking.
- Use `Get Payment` to confirm final transaction state.
- Implement proper pagination handling for production environments.
- Securely store API credentials and never expose them client-side.

## Inline Checkout

Inline Checkout lets you embed a fully managed payment experience directly inside your application — without redirecting the customer to an external page.

Your only backend responsibility is to **issue a checkout token**. Everything else — OTP verification, card setup, and payment execution — is handled by the Paywint Inline Checkout SDK on the client side.

<Steps>

  <Step title="Issue a checkout token (your backend)">
    Call **Create Developer Checkout Token** from your server, passing the payer's email or phone number, the payment amount, and currency.

    Paywint returns a short-lived `checkout_token` that encodes the payment context securely on the server side. Pass this token to your frontend.
  </Step>

  <Step title="Initialize the SDK (client side)">
    Your frontend passes the `checkout_token` to the Paywint Inline Checkout SDK to start the session.

    The SDK contacts Paywint to verify the token, identify the payer, and send them a one-time passcode (OTP) via email or SMS.
  </Step>

  <Step title="Payer verifies identity">
    The payer enters the OTP in your UI. The SDK handles verification and receives a session access token internally — your frontend does not need to manage this.
  </Step>

  <Step title="Card setup (if paying by card)">
    If the payer selects a card payment method, the SDK automatically handles any required card validation and image upload steps before proceeding to payment.
  </Step>

  <Step title="Payment is submitted">
    The SDK submits the payment to Paywint. Once processed, it returns the payment result to your frontend — including the payment ID, status, amount, currency, and payment method.

    Paywint also sends webhook notifications to your backend as the payment moves through its lifecycle.
  </Step>

</Steps>

<Info>
  The checkout token is valid for 15 minutes. Issue it as close to the moment the customer initiates checkout as possible.
</Info>
